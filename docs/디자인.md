# DayPic 디자인 개편 계획서 v2

_Author: 시니어 프로덕트 디자이너 × CTO (내부팀) / Reviewed: UX 컨설턴트 × 프론트엔드 아키텍트 (외부)_
_최종 수정: 2026-04-22_
_레퍼런스: 숨고(soomgo.com)_

## 0. Executive Summary

숨고를 참고해 **정보 위계·컬러 절제·컴포넌트 토큰화**를 도입하되, 웨딩 특유의 "무드 탐색" 행동에 맞추어 **작품이 먼저 보이는 메인**으로 재설계한다. 외부 리뷰에서 지적된 3대 리스크(① 가짜 평점 표기, ② 검색 중심 Hero가 탐색 유저를 차단, ③ 퍼널 중간 톤 불일치)를 계획 단계에서 원천 제거했다.

**1차 계획 대비 v2 주요 변경 (Top 5)**
1. "오늘의 추천 작가(랜덤 셔플)" → **"이번 주 새 포트폴리오"** (사실 기반 라벨) + 추후 "에디터 픽" 수동 큐레이션 로드맵
2. "검색 중심 Hero 4분할 pill bar" → **"Hero 모자이크 3~4컷 + 축약 검색(지역 + 스타일 태그)"**. 날짜 필드는 상세 진입 시 받음
3. Phase A(토큰/폰트) 독립 배포 폐기 → **Phase B에 Day 0로 흡수**. Phase B + Phase C(검색 카드)를 **단일 릴리스로 merge**
4. 카드에 "★ 평점(리뷰 수)" 표기 삭제. `reviews` 테이블·`review_count` 컬럼이 생기기 전까지 "포트폴리오 N장 · 지역 · 시작가"만
5. 카드 그리드 `lg:grid-cols-4` → **`lg:grid-cols-3 2xl:grid-cols-4`**로 보수화(웨딩 사진 감상 밀도)

## 왜 지금 개편하는가 (Context)

- 현재 DayPic 메인은 뉴모피즘 카드 + 다중 그라디언트 + 이모지 아이콘 + `font-black` 58px로 시각 밀도가 과하고, 정보 위계가 흐릿하다. 실제 `app/globals.css:25`는 Geist 변수를 선언해놓고 `font-family: Arial` 로 덮어써서 **한글 본문이 Arial로 렌더링되는 버그**까지 묻혀 있다.
- 컬러/반경/섀도우가 모두 JSX에 하드코딩되어 있어(`#7a5cf6`, `#7b5cf6`, `#6d46f6` 등 보라 계열만 6종 이상 산재) 토큰 리팩터 없이는 일괄 변경이 불가능.
- 숨고의 "검색 + 정보 위계 + 절제된 컬러"를 흡수하되, 웨딩 시장의 감정적 탐색 UX는 포기하지 않는 **하이브리드** 방향이 합의됐다.

**이 계획에 도달한 과정**
1. 현재 디자인 상태 실측 (메인/검색/헤더/컬러/타이포/레이아웃)
2. 숨고 홈페이지 + 스크린샷 분석으로 벤치마크 정리
3. 시니어 디자이너 × CTO 페어가 1차 계획 수립
4. 외부 UX 컨설턴트 × 프론트엔드 아키텍트가 공격적 리뷰 (14 크리틱 + Verdict 3대 철회 권고)
5. 주요 리뷰어 주장(`review_count` 부재, checklist/tips 페이지 실존, globals.css 폰트 버그)을 **직접 코드에서 검증**
6. v2로 수정·확정

## 3. 타이포그래피

**폰트**: Geist → **Pretendard Static Subset (Bold / SemiBold / Regular)**.
- `next/font/local` 로 woff2 3종만 로드 (~80KB), `display: swap`, `preload: true`.
- Variable 원본 ~1.1MB은 회피. Subset(korean)만.
- CSS `size-adjust` / `ascent-override` 디스크립터로 Geist→Pretendard 스왑 시 CLS를 0에 근접시킴.

**반드시 수정 2곳**:
- `app/globals.css:25` — `font-family: Arial, Helvetica, sans-serif;` **즉시 삭제** (Geist 무력화의 범인)
- `app/layout.tsx:2,6-14` — Geist/Geist_Mono 제거, Pretendard 로컬 로더로 교체

**스케일**
| 토큰 | 클래스 | 용도 |
|---|---|---|
| H1 Hero | `text-[44px] md:text-[52px] font-bold tracking-[-0.03em] leading-[1.15]` | 랜딩 메인 타이틀 |
| H2 Section | `text-[26px] md:text-[30px] font-bold tracking-[-0.02em]` | 섹션 제목 |

## 4. Hero 리디자인 (v2 — 무드 우선)

**철학 변경**: 1차안의 "검색 바 중심 Hero(4분할 pill bar)"는 웨딩 유저의 초기 탐색 행동(지역·날짜 미확정인 무드 스크롤)과 충돌. v2는 **대표 작품 모자이크 + 축약 검색**.

```
┌────────────────────────────────────────────────────────────┐
│                                                            │
│  ┌──────┐  평생 한 번, 인생에 남는                          │
│  │      │  사진을 찾는 가장 빠른 길.                         │   <- h1 44~52px bold
│  │      │                                                   │
│  │ MOOD │  ┌─────────────────────────────┐                 │
│  │ 모자이크│  │ [지역 ▼]  [#본식스냅 #야외…] │ [검색] │        │   <- 56px 둥근 pill
│  │ 3~4컷 │  └─────────────────────────────┘                 │
│  │      │                                                   │
│  │      │  인기:  #본식스냅  #야외촬영  #서브스냅            │
│  └──────┘                                                   │
│                                                            │
│  (스크롤 힌트 ↓  다음 섹션에 카테고리와 최신 포트폴리오)     │
└────────────────────────────────────────────────────────────┘
배경: #FFFFFF. 그라디언트·뉴모피즘 없음.
```

- Hero 높이: 데스크탑 `min-h-[560px]`, 모바일 `min-h-[540px]` (55vh 이상 금지 — 첫 스크롤에 다음 섹션이 보여야 함)
- 좌측 무드 모자이크: `grid grid-cols-2 grid-rows-2 gap-2 aspect-[4/5]` 에 대표 포트폴리오 4컷. 이미지 소스는 Phase 1에서 에디터가 수동 지정(env 또는 `featured_hero_images` 테이블)
- 검색 바: `flex items-center h-14 rounded-full border border-border-default bg-white px-2 pl-5 max-w-[520px] shadow-[0_2px_8px_rgba(0,0,0,0.04)]`
  - 지역 드롭다운 + 스타일 태그 멀티셀렉트(최대 2개) + 검색 버튼
  - **날짜는 Hero에서 제거**. 상세 페이지의 "예약 가능 여부 확인" 단계에서 받음
- 인기 검색 chip: `inline-flex h-8 items-center rounded-full border border-border-default bg-white px-3 text-[13px] text-fg-muted hover:border-primary hover:text-primary`

**모바일**
- 모자이크와 타이틀 순서 반전: **무드 모자이크 → H1 → 축약 검색(단일 필드) → 인기 검색 chip**
- 검색은 한 줄 `placeholder="지역·스타일로 찾기"` 단일 입력, 탭 시 하단 시트 전개
- Sticky 적용 여부는 Phase 3에서 A/B 테스트 후 결정 (현재 정적)

## 8. 썸네일 전략 (4:5 전환 + 수동 대표 이미지)

1차안 Cloudinary `g_face` 자동 크롭은 **2인 풀샷/뒷모습/풍경에서 비대칭 크롭 실패**. v2는 명시적 수동 지정 모델.

- `artists` 테이블에 `main_image_url TEXT` 컬럼 추가(마이그레이션 007)
- 작가 편집 화면(`app/artist-profile/edit`)에 "카드 대표 이미지 지정" 섹션 신설 — 포트폴리오에서 1장 선택
- 카드 렌더링 우선순위: `main_image_url` → 포트폴리오 첫 장 → Unsplash 폴백
- Cloudinary 변환 헬퍼 `lib/cloudinary.ts`의 `buildThumbnailUrl(url, { ar: '4:5', width: 600 })`:
  - 지정된 대표 이미지: `c_fill,ar_4:5,w_600,q_auto,f_auto` (gravity는 center — 작가가 이미 구도 확정)
  - 폴백 이미지: `c_fill,ar_4:5,g_auto,w_600,q_auto,f_auto` (g_auto = Cloudinary saliency)

**런칭 전 대표 이미지 점검 스프린트 0.5일** 필수 포함. 런칭 후 "사진이 이상하게 잘려요" CS를 사전 차단.

## 9. Header 개편

**1차안 4항목 → v2 3+1 구조**: 좌측 네비(신부 저니) + 우측 독립 CTA(작가 진입) 분리.

```
[로고] ────── [작가 찾기] [체크리스트] [팁] ──────── [📍 지역] [작가로 활동하기] [로그인]
```

- **좌측 네비** (신부 대상): "작가 찾기"(`/search`) / "체크리스트"(`/checklist`) / "팁"(`/tips`)
- **우측 분리 CTA** (작가 대상): "작가로 활동하기"(`/artist-register` 또는 `/artist-dashboard` — 로그인 상태 분기). 텍스트 링크 스타일, 보더 없음, `text-[13px] text-fg-muted hover:text-primary`
- 지역 드롭다운: 데스크탑만 노출. localStorage에 저장 → 검색 기본값 프리필 (Phase 2)
- 모바일: `lg:hidden` 햄버거 → 풀스크린 시트

## 14. 검증 / 테스트 플랜 (end-to-end)

1. **로컬 시각 검증**
   - `npm run dev` (포트 3001) — 메인 Hero에 작품 모자이크 4컷이 노출되는가
   - 콘솔에 Cloudinary 이미지 403/404 없는가
   - `document.body.style.fontFamily` 가 Pretendard인지 (Arial 아님)
2. **반응형**
   - 360 / 768 / 1024 / 1440 / 1920px 스냅샷 비교
   - 모바일 Hero 첫 스크롤에 카테고리 섹션 일부가 비치는가 (55vh 제약 확인)
3. **기능 회귀**
   - 카테고리 클릭 → `/search?service=본식스냅` 직행 + 필터 프리필
   - 찜/최근본작가 localStorage 플로우 정상
   - 검색 → 카드 → 상세 → 포트폴리오 뷰어 경로 끊김 없음
4. **데이터 진실성**
   - 카드에 별점/리뷰 수 표기 **없음** 확인
   - "이번 주 새 포트폴리오"가 `created_at DESC` 기준인지 네트워크 탭에서 응답 검증
5. **접근성**
   - Hero 검색 바 키보드 탐색(Tab) 순서 정상
   - 카테고리 카드 `<a>` 태그 포커스 링 노출
   - 모바일 햄버거 `aria-expanded` 토글
6. **성능**
   - Lighthouse LCP < 2.5s, CLS < 0.05
   - Pretendard woff2 단일 요청 < 100KB

## 16. 오픈 이슈 (의사결정 필요)

1. **Pretendard 로컬 번들 vs CDN** — 라이선스상 둘 다 가능. 번들 권장.
2. **"작가로 활동하기" 분리 CTA의 데스크탑 위치** — 우측 로그인 좌측 텍스트 링크 vs 별도 아이콘 버튼.
3. **Phase 0 `reviews` 테이블 설계 시작 시점** — Phase 1 착수 전 vs Phase 1 병렬.
4. **Hero 모자이크 4컷 선정 메커니즘** — 환경변수 하드코딩(v1) vs `featured_hero_images` 테이블(v2) — 초기에는 v1, 운영팀 생기면 v2 승격.
