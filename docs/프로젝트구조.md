# DayPic 프로젝트 구조

> 날짜 기반 웨딩 촬영 작가 매칭 플랫폼.
> 신랑·신부와 고객은 예식 날짜·지역·서비스 기준으로 작가를 검색하고, 작가는 본인의 포트폴리오·휴무일·영상을 관리한다. **사이트는 동시에 작가의 포트폴리오 사이트 역할도 한다** (양면 동등 노출 원칙).
>
> 본 문서는 **현재 코드의 사실관계 + Supabase 스키마 + API 계약**을 기록한다.
> UI/UX·레이아웃·상태 매트릭스·Sprint 백로그는 [`docs/디자인.md`](디자인.md) v4를 참조한다.
>
> 마지막 갱신: 2026-04-23 (v4 정합)

## API 라우트

> v4 응답 envelope 표준 ([디자인.md §26](디자인.md)):
> ```
> { ok: boolean, dbError?: boolean, error?: { code, message, fields? }, ...payload }
> ```
> 현재 일부 엔드포인트는 `success` 키 잔존(중복) — Sprint 4에서 폐기 예정.

### 검색

#### `GET /api/search`
**Query**: `date`, `region`, `service` (복수), `price`
**동작**: Supabase artists `service_role` 조회 → 인메모리 필터 → closed_dates 제외 → shuffle
**응답**: `{ ok, artists[], total, dbError? }` (v4 envelope)
**파일**: `app/api/search/route.ts`

#### `GET /api/artists/showcase` *(NEW v4, Sprint 2)*
**Query**: `limit=10` (default 10)
**동작**: weighted_random `1 / (epoch(now() - last_shown_at) + 1)` → 노출된 10명에 `last_shown_at = now()` 갱신
**ISR**: `next.revalidate: 300` (5분)
**의존**: 마이그레이션 009 (`artists.last_shown_at`)
**파일**: `app/api/artists/showcase/route.ts` *(미구현)*

### 작가 관리

#### `GET /api/artists/[id]`
4종 식별자 분기 (`lib/artist-lookup.ts`). 응답에 `updated_at` 포함 (낙관적 락 토큰).
응답 형식:
```
{
  ok: true,
  ...artist,
  video_portfolio_items: [{ position, link, thumb, style_tags[] }],
  // 레거시 필드 (deprecated, 클라 전환 후 제거):
  video_link_1..4, video_thumb_1..4, video_thumbnail, video_style_tags,
  keywords, 성향키워드, // → style_keywords 단일화 완료, 응답에는 호환용 잔존
  updated_at,
  dbError?
}
```

#### `PATCH /api/artists/[id]`
**Body**: 일부 필드 (`phone`, `price`, `service[]`, `region[]`, `style_keywords[]`, `testimonials[]`(S5), `expected_updated_at?`)
**낙관적 락**: `expected_updated_at` 동봉 시 DB 값과 비교 → 불일치면 `409: { ok:false, error:{ code:OPT_LOCK, current_updated_at } }`
**권한**: `requireArtistOwner` (session.kakaoId ↔ artist.kakao_id, 레거시 email 폴백)

#### `POST /api/artists/register`
**Body**: `{ companyName, email, phone, services[], regions[], price, styleKeywords[], portfolioUrl, openchatUrl }`
**응답**:
- 200: `{ ok, artist }`
- 400: `{ ok:false, error:{ code:VALIDATION, fields:{...} } }`
- 409: `{ ok:false, error:{ code:UNIQUE_KAKAO | UNIQUE_EMAIL } }`

#### `GET /api/artists/by-email`
이메일로 조회. *Sprint 4에서 success 키 폐기 예정.*

#### `PATCH /api/artists/[id]/portfolio-images`
**Body**: `{ imageUrls: string[] }` (최대 40)

#### `PATCH /api/artists/[id]/representative-image`
**Body**: `{ imageUrl: string }`
**현재 응답**: `{ ok, success, record:{ id, fields:{ "대표사진":[{ url }] } }, imageUrl }` *(Airtable shape leak)*
**v4 (Sprint 4) 정규화**: `{ ok, imageUrl, dbError? }`

#### `PATCH /api/artists/[id]/video-portfolio`
**Body**: `{ video_link_1..4, video_thumb_1..4, video_style_tags }`
**동작**: 직접 `video_portfolio_items` upsert + `artists.updated_at` 갱신 (낙관적 락 토큰)
**현재 응답**: `{ ok, success, message, recordId }` — *Sprint 4에서 success 키 폐기*

#### `GET /api/artists/[id]/stats` *(NEW v4, Sprint 5)*
**응답**: `{ ok, week_impressions, week_card_clicks, total_favorites, closed_dates_count, dbError? }`
**파일**: `app/api/artists/[id]/stats/route.ts` *(미구현)*

### 일정 관리 (`/api/artist-closed`)

#### `GET`
응답: `{ ok, dates: ["YYYY-MM-DD"...], records: [{ id, date, artist_id }...] }`

#### `POST`
**Body**: `{ date }` — 중복은 idempotent (`{ ok, duplicated:true, ... }` 200)

#### `DELETE`
**Query**: `?date=YYYY-MM-DD` — 미존재도 idempotent (`{ ok, already_deleted:true, ... }` 200)

### 인증

#### `GET & POST /api/auth/[...nextauth]`
NextAuth 카카오 OAuth, JWT 전략. 콜백:
1. `signIn` — kakao 인증 정보 로깅
2. `jwt` — kakaoId 추출 → Supabase `users` upsert + `artists` 조회 → token에 `userId`/`artistId`/`kakaoId`/`dbError` 추가
3. `session` — token 값을 session 객체에 전달

#### `GET /api/me`
현재 세션 정보. 응답: `{ ok, isLoggedIn, isArtist, userId, artistId, artistCode, kakaoId, email, name, dbError }`

### 파일 업로드

#### `POST /api/cloudinary/sign`
업로드 서명 발급. **작가 세션 필수**, **folder allowlist** (현재: `daypic/artists/representative`).
서명: `SHA1(folder={folder}&timestamp={timestamp}{apiSecret})`.
클라이언트가 직접 Cloudinary로 업로드.

### 시스템

#### `GET /api/health` *(2026-04-21 신설)*
Supabase 연결 + 주요 테이블 ping.
응답: `{ ok, env, checks:{ artists:{ ok, latency_ms, error? }, closed_dates:{...}, video_portfolio_items:{...} }, duration_ms }`
정상이면 200, 실패면 503.

## Supabase 스키마

> Airtable은 Phase 3에서 완전 폐기. 운영 코드는 Supabase 단독 (`@supabase/supabase-js` + `service_role`).
> `scripts/airtable-helpers.ts` 와 `scripts/migrate-airtable-to-supabase.ts` 는 1회 이관용으로만 잔존.

### `artists` 테이블

PRIMARY KEY: `id` (Airtable 호환 `recXXX` 포맷, 모든 FK 기준)

주요 컬럼:
- `id` — PK, `recXXX` (legacy from Airtable)
- `artist_id` — UNIQUE, 자체 코드 `artist_xxx`
- `user_id` → `users(id)` FK (ON DELETE SET NULL)
- `kakao_id` — UNIQUE, 카카오 숫자 ID
- `name`
- `email`, `phone`
- `service` text[]
- `region` text[]
- `price` text — 자유 입력 *(deprecated; 마이그레이션 008에서 `price_min INT`/`price_max INT` 분리 예정)*
- `style_keywords` text[] *(`keywords` alias deprecated, 응답에만 잔존)*
- `portfolio` (URL)
- `image` — 대표 이미지 URL *(주: v4에서 `main_image_url` 별칭 추가 검토)*
- `portfolio_images` text[] (최대 40)
- `open_chat_url`
- `artist_type`
- `rating` numeric *(default 4.8 → 마이그레이션 008에서 NULL 허용으로 변경 예정, 데이터 정직 원칙)*
- `created_at`, `updated_at` (트리거 자동 갱신, 낙관적 락 토큰)

**Phase 4.2 Contract (마이그레이션 007, 2026-04-23 적용)**:
- DROP: `video_link_1..4`, `video_thumb_1..4`, `video_thumbnail`, `video_style_tags`
- 단일 진실원은 별도 `video_portfolio_items` 테이블

**v4 신설 예정 컬럼**:
- 008: `testimonials JSONB DEFAULT '[]'`, `price_min INTEGER NOT NULL DEFAULT 0`, `price_max INTEGER`, `rating` DEFAULT 제거 → NULL 허용
- 009: `last_shown_at TIMESTAMPTZ NULL` + 부분 인덱스 (메인 §6.0 쇼케이스 회전용)

### `users` 테이블

- `id` PK
- `kakao_id` UNIQUE

- `created_at`, `updated_at`

anon SELECT는 마이그레이션 001/004에서 PII 보호 위해 차단.

### `closed_dates` 테이블

- `id` PK
- `artist_id` → `artists(id)` FK (ON DELETE CASCADE)
- `closed_date` (YYYY-MM-DD)
- UNIQUE(`artist_id`, `closed_date`) 인덱스 (마이그레이션 004)

`artist_id`는 반드시 recordId 포맷 (`recXXX`). 마이그레이션 001/003에서 혼종 ID 정리됨.

### `video_portfolio_items` 테이블 (마이그레이션 005 신설, 007에서 단일 진실원으로 승격)

- `id` PK

- `link` text
- `thumb` text NULL
- `style_tags` text[] NULL
- `created_at`, `updated_at`
